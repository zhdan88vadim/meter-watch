## What is Redis?

Redis is an **in-memory data structure store** used as a database, cache, message broker, and streaming engine. It's extremely fast because it keeps data in RAM, and it supports various data structures:

- **Strings** – basic key-value
- **Lists** – ordered collections (like queues)
- **Sets** – unique, unordered collections
- **Sorted Sets** – sets with scores for ordering
- **Hashes** – field-value maps (like objects)
- **Streams** – append-only logs (like Kafka)
- **Pub/Sub** – real-time messaging
- **Geospatial** – location-based data

### 1. `publish()` – Real-time Broadcasting

| Aspect | Detail |
|--------|--------|
| **Method** | `publish(channel, message)` |
| **What it does** | Sends a message to all subscribers of that channel |
| **Data structure used** | Pub/Sub (not stored in memory) |
| **Use case** | Live notifications, WebSocket backends, real-time dashboards |
| **Important** | Messages are **not persisted**. If no one is listening, the message is **lost forever**. |

**Example subscribers:**
```python
# In another process/service
pubsub = r.pubsub()
pubsub.subscribe('detection:events')
for message in pubsub.listen():
    if message['type'] == 'message':
        data = json.loads(message['data'])
        print(f"Live alert: {data}")
```

---

### 2. `lpush()` + `ltrim()` – History Storage

| Aspect | Detail |
|--------|--------|
| **Method** | `lpush(key, value)` – adds to the **left** (head) of a list |
| **Method** | `ltrim(key, start, stop)` – keeps only the specified range |
| **Data structure used** | **List** (ordered, stored in memory) |
| **Use case** | Recent history, activity logs, last-N items |
| **Important** | Data **is persisted** (until memory is evicted or you delete it) |

**What the combination does:**
```
lpush → [event3, event2, event1]  (newest at index 0)
ltrim → keeps indices 0 to 999 → max 1,000 events
```
So you keep **only the 1,000 most recent** events automatically.

**Retrieving history:**
```python
# Get all 1000 events
history = r.lrange('detection:history', 0, -1)

# Get only last 10
recent = r.lrange('detection:history', 0, 9)
```

## When to Use What – Decision Guide

### Use `publish()` when:

| Scenario | Example |
|----------|---------|
| You need **instant notifications** | Live alerts, chat messages, stock price updates |
| Subscribers are **ephemeral** | Web frontends, mobile push, server-sent events |
| You don't care about **missed messages** | If a client disconnects, old messages don't matter |
| You want **low latency** | Pub/Sub is extremely fast (~microseconds) |
| You have **many consumers** | One publisher, many subscribers |

### Use `lpush()` + `ltrim()` when:

| Scenario | Example |
|----------|---------|
| You need **recent history** | Last 100 detection events, recent user actions |
| Clients **reconnect and need past data** | Dashboards that reload |
| You want a **fixed-size log** | Keep only the last N items automatically |
| You need **ordered data** | Timeline view (newest first) |
| You want **persistence** | Even if no client is listening, data is stored |

## When to Use Other Redis Structures

| Structure | Best For |
|-----------|----------|
| **Strings** | Caching API responses, storing session tokens, counters (`incr`) |
| **Hashes** | Storing user profiles, settings, objects |
| **Sets** | Unique tags, followers, friend lists, deduplication |
| **Sorted Sets** | Leaderboards, priority queues, time-based ordering |
| **Streams** | Event sourcing, message queues with consumer groups (more reliable than Pub/Sub) |
| **Geo** | Nearby stores, location-based search |

## Full Pattern – Your Code + Consumer

```python
import redis
import json
import time

r = redis.Redis(decode_responses=True)

# --- Producer ---
def send_detection(event):
    # 1. Real-time notification
    r.publish('detection:events', json.dumps(event))
    
    # 2. Store history (keep last 1000)
    r.lpush('detection:history', json.dumps(event))
    r.ltrim('detection:history', 0, 999)

# --- Consumer: Real-time ---
def listen_live():
    pubsub = r.pubsub()
    pubsub.subscribe('detection:events')
    for msg in pubsub.listen():
        if msg['type'] == 'message':
            print(f"🔥 LIVE: {msg['data']}")

# --- Consumer: History on demand ---
def get_recent_history(limit=50):
    return [json.loads(item) for item in r.lrange('detection:history', 0, limit-1)]

# --- Example ---
send_detection({"id": 123, "type": "motion", "timestamp": time.time()})
print(get_recent_history(5))
```

## Pro Tips

1. **Use `decode_responses=True`** – automatically converts bytes to strings
2. **Use pipelines** for batch operations:
   ```python
   pipe = r.pipeline()
   pipe.lpush('history', data)
   pipe.ltrim('history', 0, 999)
   pipe.execute()  # sends both commands at once
   ```
3. **Set TTL (expiry)** if you want old data to auto-delete:
   ```python
   r.expire('detection:history', 3600)  # auto-delete after 1 hour
   ```
4. **Consider Redis Streams** if you need:
   - Acknowledged processing (consumer groups)
   - Message persistence
   - Guaranteed delivery
   - > Pub/Sub is fire-and-forget, Streams are more like Kafka
